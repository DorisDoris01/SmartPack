Part 1 — GeneralWhat to Improve

1. No CLAUDE.md in the project itself

Your SmartPack repo has no CLAUDE.md. This is the single highest-impact thing you're missing. Every session, Claude Code starts cold on your project conventions. You should have a CLAUDE.md in the project root with: build command, test command, architecture overview (MVVM, SwiftData, bilingual approach), design system file locations, and key constraints.

The memory file helps, but CLAUDE.md is loaded with higher priority and is project-scoped rather than directory-scoped.

2. Planning docs accumulating in repo root

You have 6+ .md files sitting in the project root alongside source code. These are working documents, not shipping artifacts. Consider moving them to a docs/ or .claude/plans/ directory, or .gitignore-ing them. Some (like pending-issues-guide.md at 56KB) are enormous and probably shouldn't be committed.

3. No branching strategy for Claude Code work

All 30+ commits are on main or develop with no feature branches for Claude Code work. The MapKit autocomplete change, the design system overhaul, and the bug fixes are all mixed together. Feature branches would give you: ability to review before merging, easy rollback, and cleaner history.

4. Permission allowlist is too broad

settings.local.json allows Bash(rm:*) globally. This gives Claude Code unrestricted rm access anywhere on your system. Consider scoping it to your project directory, or removing it and approving on a case-by-case basis.

5. Testing came too late

Test coverage (F5) was listed as Critical but was one of the last things fixed. You had 30+ commits of feature work and bug fixes before adding any tests. TDD or "write tests alongside fixes" would catch regressions earlier.

6. Prompt specificity could improve

Some planning docs hint that you rely on Claude to audit everything rather than pointing it at specific concerns. Focused prompts ("audit the touch targets in PackingListView against Apple HIG 44pt minimum") get more precise results than broad ones ("do a UX audit").

7. No verification step in workflow

Several fixes had to be rolled back (U5→U8, A2→U2). Building a habit of "verify on device before committing" would reduce churn.

  • Explore agents dominate — used in nearly every session. Good instinct to let agents explore the codebase rather than doing it manually.

What's missing

No parallel agent dispatching

When you had the dual code review, those two reviewers could have run in parallel in isolated worktrees. When you had multiple independent UX audits (HomeView, PackingListView, ItemManagement), those could have been 3 parallel agents. You're running things sequentially that don't need to be.

Worktrees barely used

Only 2 sessions referenced worktrees. For a project where you're doing design overhauls, bug fixes, and feature additions on main, worktrees would let you experiment safely. The MapKit autocomplete feature (5 files, new service + view) was a perfect worktree candidate.

No subagent-driven-development

When you have a multi-step plan (like the 5-step MapKit implementation plan), the subagent-driven-development skill dispatches independent steps to parallel agents. You wrote the plan manually and executed it sequentially.
