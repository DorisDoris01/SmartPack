Drill-Down City Search List

Can we add an autocomplete dropdown when typing a destination? Here's every option evaluated with effort, pros, and cons.

1

What We're Evaluating

The Idea

Instead of a plain text box where you type a city and hope for the best, we'd add a search-as-you-type dropdown. As you type, a list of matching cities appears below the text field. You tap the correct one, and the app knows exactly which city you mean — no more guessing, no more wrong results.

This solves all the problems from the investigation: "赣州" would show "赣州市, 江西, 中国" in the list. "Lombok" would show "Lombok Island, Indonesia" alongside the small villages, so you pick the right one.

Five Approaches Compared

There are different ways to power the suggestion list. Each uses a different data source to look up city names. Here they are from simplest to most thorough.

A
Open-Meteo API Autocomplete
Use the same API we already have, but show multiple results
Not Recommended
Development Effort ~2 days

How It Works

As you type, the app calls the same Open-Meteo geocoding API but with count=5 to get multiple results. Shows them in a dropdown so you pick the right one.

Pros
  • Least code to write — reuses existing API
  • No new dependencies or permissions
  • Already free, no API key needed
Cons
  • Doesn't fix the root problem! "赣州" still returns 0 results from this API
  • Chinese names for foreign places still missing (龙目岛, 巴厘岛)
  • Results have no autocomplete/fuzzy matching — you need to type the full name
  • Needs internet for every keystroke

Files Affected

  • NEW Components/Trip/CitySearchField.swift
  • MODIFY Components/Trip/CreateTripSheet.swift
  • MODIFY Services/WeatherService.swift (expose geocoding)
B
Apple MapKit Search Completer
Use Apple's built-in search-as-you-type for locations
Recommended
Development Effort ~3 days

How It Works

Apple has a built-in tool called MKLocalSearchCompleter that's specifically designed for "search as you type" location lookups — it's the same technology that powers the search bar in Apple Maps. It understands Chinese, English, and every other language that Apple Maps supports.

When you type "赣州", it instantly suggests "赣州市, 江西省, 中国". When you type "龙目", it suggests "龙目岛, 印度尼西亚". When you type "Bali", it shows "Bali, Indonesia" at the top. It handles all the edge cases we found.

No location permission needed. It only searches a database of place names — it doesn't track where you are.

Pros
  • Fixes all reported bugs — "赣州", "龙目岛", "巴厘岛" all work
  • Excellent Chinese language support (it's Apple Maps data)
  • True autocomplete — shows results as you type each character
  • Knows which city is more popular (Beijing the capital, not the village)
  • No location permission needed — won't scare users
  • Free and built into every iPhone — no API key, no usage limits
  • Already familiar UX — works like Apple Maps search
Cons
  • Needs internet (but so does weather fetching anyway)
  • Adds MapKit framework dependency (~1 MB)
  • Still need Open-Meteo for the actual weather data — need to convert Apple's result to coordinates
  • Apple controls the data — can't customize ranking
💡

Key insight: MKLocalSearchCompleter gives us coordinates directly in the search result. So we can bypass the Open-Meteo geocoding step entirely — go straight from the user's selection to fetching weather with coordinates. This eliminates all geocoding bugs in one move.

Files Affected

  • NEW Services/LocationSearchService.swift (~80 lines)
  • NEW Components/Trip/CitySearchField.swift (~120 lines)
  • MODIFY Components/Trip/CreateTripSheet.swift (swap TextField)
  • MODIFY Services/WeatherService.swift (accept coordinates)
C
Local Curated City Database
Ship a JSON file of popular cities with the app
Viable Alternative
Development Effort ~3–4 days

How It Works

Bundle a JSON file inside the app containing ~500–1000 popular travel destinations worldwide, each with Chinese name, English name, and coordinates. When you type, the app searches this local list instantly — no internet needed.

Think of it like a phone book that comes pre-loaded with the most popular cities. If your city isn't in the book, you can still type it manually and it falls back to the API.

Pros
  • Works offline — search works even without internet
  • Instant response — no network delay
  • Full control over Chinese names (龙目岛, 巴厘岛, 普吉岛 etc.)
  • Can prioritize cities popular with Chinese travelers
  • No permissions needed, no external dependencies
Cons
  • Limited coverage — can only search cities in the list
  • Needs manual curation and maintenance
  • Obscure destinations will be missing
  • Building the initial dataset takes effort (research 500+ cities)
  • Increases app bundle size (~50–100 KB)

Files Affected

  • NEW Resources/cities.json (~500 entries)
  • NEW Services/CityDatabase.swift
  • NEW Components/Trip/CitySearchField.swift
  • MODIFY Components/Trip/CreateTripSheet.swift
  • MODIFY Services/WeatherService.swift
D
Apple CLGeocoder
Use Apple's geocoding to replace Open-Meteo geocoding
Overkill
Development Effort ~3–4 days

How It Works

Apple's CLGeocoder converts place names to coordinates (like Open-Meteo's geocoding, but much better for Chinese). Instead of adding a dropdown, this replaces the behind-the-scenes name lookup while keeping the same text field UI.

Pros
  • Excellent Chinese support — handles all edge cases
  • Minimal UI changes — same text field, just better backend
  • Free, built into iOS
Cons
  • No autocomplete UI — doesn't add a drill-down list
  • Still vulnerable to ambiguous names (user types "Bali", gets... what?)
  • Rate-limited — only 1 request at a time (not for live search)
  • Needs internet
  • Fixes backend but doesn't improve user experience

Files Affected

  • MODIFY Services/WeatherService.swift (replace geocoding logic)
  • MODIFY Info.plist (maybe, for location usage description)
E
Hybrid: Local Database + MapKit
Best of C and B — local first, Apple as backup
Gold Standard
Development Effort ~5–6 days

How It Works

Combine a local database of popular cities with Apple's MapKit search. When you type, the app first checks the local list for instant results, then also queries Apple for additional matches. Results are merged and deduplicated.

It's the "ultimate" solution, but at the cost of more code to build and maintain.

Pros
  • Popular cities show up instantly (offline)
  • Apple fills in the gaps for obscure destinations
  • Best possible coverage and speed
Cons
  • Most complex to build — nearly double the effort of Option B
  • More code to maintain long-term
  • Merging two data sources adds edge cases
  • May be over-engineering for a packing app

The Verdict

Recommendation: Option B — Apple MapKit Search Completer

Why? It's the sweet spot between effort and quality. In ~3 days of development:

1. It fixes every single bug we found — "赣州", "北京", "龙目岛", "Lombok", "Bali" — all resolved.
2. It gives users a polished drill-down list that feels like Apple Maps search.
3. It needs no permissions, no API key, no usage limits, and no maintenance.
4. It covers every city in the world, not just a curated list.
5. It bypasses Open-Meteo's buggy geocoding entirely — MKLocalSearchCompleter gives coordinates directly, so we skip the broken name-to-coordinates step altogether.

Option A doesn't fix the core problems. Option C is good but limited to pre-loaded cities. Option D doesn't add the drill-down list. Option E is the "gold standard" but costs nearly double the effort for marginal improvement over B alone. Option B gives you 95% of the value at 50% of the maximum cost.

🛠

Implementation path for Option B:

1. Create LocationSearchService — thin wrapper around MKLocalSearchCompleter
2. Create CitySearchField — new UI component with TextField + suggestion dropdown (can follow the existing pattern from AddItemRow suggestions)
3. Swap into CreateTripSheet — replace the plain TextField with the new component
4. Update WeatherService — add a method that accepts coordinates directly (skip geocoding)

What if you want to start even smaller?

If 3 days feels like too much right now, there's a "quick patch" you can apply in ~30 minutes that doesn't add a drill-down list but fixes the worst bugs: auto-append "市" for 2-character Chinese city names, and request count=5 instead of count=1 and pick the result with the highest population. This won't fix 龙目岛 or 巴厘岛, but it fixes 赣州, 北京, 成都, Bali, and most ambiguous names. You can always add the drill-down list (Option B) later.
